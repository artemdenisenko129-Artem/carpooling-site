# План розвитку сайту «Попутки UA»

Документ-нотатка. Єдина цілісна картина: концепція, що вже зроблено,
як воно поєднується з тим, що треба зробити, і в якому порядку.
Оновлюється по ходу роботи.

---

## 1. Концепція сервісу

- Це сервіс для КОРОТКИХ ЩОДЕННИХ ПРИМІСЬКИХ поїздок: дім — робота — дім.
- НЕ міжміські далекі поїздки, а приміські напрямки навколо міста.
- Аудиторія локальна — села, міста-супутники, ЖК, райони.
- Ключова проблема: пошук за точним збігом назв НЕ працює.
  Приклад: водій їде «Димер → Київ», пасажир шукає «Петрівці → Київ».
  Петрівці лежать на дорозі водія, але назви не збігаються.
- Рішення: маршрут + КАРТА для візуального пошуку.
  Людина сама бачить на карті, хто проїжджає поряд із нею.
- Автоматичний матчинг НЕ робимо — людина дивиться карту і вирішує сама.

## 2. Архітектура — Шлях 2 (сайт головний)

- Сайт — головний інструмент: пошук, перегляд, створення оголошень.
- Бот — лише невидима технічна сутність для входу через Telegram.

- Один проєкт, безкоштовний хостинг (Vercel).

## 6. ФАЗА 1 — ядро (рекомендований порядок виконання)

1. Світлий мобільний дизайн (mobile-first), без пафосу й лендинг-секцій.
2. Власний реєстр населених пунктів України + автокомпліт (розділ 8).
3. Карта (Leaflet + OSM) — показ маршрутів, маркери (розділ 7).
4. Форма оголошення: маршрут, дата/розклад, роль, телефон

   Ввід точок — автокомпліт АБО точка на карті.
5. Сторінка каталогу з перемикачем «Список / Карта» (розділ 7).
6. Фільтри: роль (усі / водій / пасажир); тип (усі / разова / періодична);

7. Вхід: Telegram-ідентифікація + Google OAuth (через Auth.js).
8. markercluster — групування маркерів (простий плагін, ставимо одразу).
9. Публікація одночасно на сайт і в канал (уже працює).
10. Базове СЕО + перелінковка (розділ 14).

## 7. Карта, стрічка і маркери — деталі (Фаза 1)

Сторінка каталогу — ОДНА, з перемикачем виду «Список / Карта»:
- Фільтри спільні, діють на обидва види одночасно.
- Карта завантажується лише при перемиканні на вид «Карта» (lazy).
- Зручно на телефоні: один тап — і потрібний вид.

Маркери:
- Три типи точок: старт / проміжні / кінцева (призначення).
- Неактивні маркери — сірі (спокійний загальний фон карти).
- При наведенні/натисканні маршрут підсвічується:
  старт і проміжні — сині, призначення — червоні.
- Форма маркерів різна: призначення — крапля; старт і проміжні —
  інша форма (кружок). Старт і проміжні однієї форми, бо проміжна
  для когось — це старт.
- Лінія маршруту — проста пунктирна, бліда, з'являється при активації.
  Орієнтир напрямку, не точний шлях. БЕЗ OSRM/маршрутизації
  (складно, ресурсомістко і дає оманливу точність).

Примітка: остаточні рішення по розділах 7-8 ще обговорюються,
це робочий опис, не зафіксована постанова.

## 8. Геокодинг і список населених пунктів (Фаза 1)

Проблема: короткий список міст когось не міститиме — це дратує
і шкодить на старті просування.

Рішення — власний ПОВНИЙ реєстр населених пунктів України:
- Усі міста, селища і села — близько 28-30 тисяч. Список кінцевий.
- Дані (назва + координати + область) — з відкритих джерел безкоштовно:
  OpenStreetMap, GeoNames, державний класифікатор КАТОТТГ.
- Автокомпліт — через власний серверний endpoint: вводиш кілька літер,
  сервер віддає 5-10 збігів. Клієнт не вантажить весь список.
- Координати міста вже в реєстрі — окремий прямий геокодинг не потрібен.

Чому власний реєстр, а не сторонні сервіси (Photon, LocationIQ):
- Не залежить від чужого сервера — не впаде, не залімітує.
- На старті просування поганий перший досвід шкодить найбільше.

Якщо населеного пункту все ж нема (рідкісна точка):
- Підказка в полі: «Почни вводити назву або постав точку на карті».
- Точка на карті → зворотний геокодинг Nominatim визначить назву.

ОТГ (територіальні громади) — не для маршруту, а для «Моєї спільноти».
Nominatim — безкоштовний, але публічний сервер має ліміти
(1 запит/сек, автокомпліт заборонено) — тому лише зворотний геокодинг.

## 10. Принципи розробки

### 10.1. Архітектура коду — кожна логіка в одному місці

Головний принцип: якщо логіка в одному місці — зміна = правка в одному
файлі. Якщо розмазана — зміна перетворюється на полювання по коду.

1. Типи даних — один файл (тип Announcement). Решта коду імпортує його.
   Додав поле — TypeScript сам підсвітить, де ще треба доробити.
2. Доступ до бази — окремий модуль із функціями (getAnnouncements,
   createAnnouncement тощо). Сторінки й API не пишуть запити напряму.
3. Зовнішні інтеграції — кожна окремим модулем (Telegram, геокодинг,
   карта). Заміна сервісу = правка одного модуля.
4. UI — дрібні перевикористовувані компоненти (AnnouncementCard
   окремо — потрібен на головній, сторінці маршруту, у попапі карти).
5. Константи (назви бази, колекцій, ліміти) — один файл конфігу.

Не переінженерити: цих принципів достатньо, складні патерни не потрібні —
вони додають заплутаності більше, ніж гнучкості.

### 10.2. Дані і схема

- Закладати поля наперед — у тип Announcement одразу: координати
  (формат GeoJSON), точки маршруту, telegramUserId, views, createdAt,
  isActive, тип поїздки (разова/періодична), community, контакти
  (telegram + телефон). Навіть якщо поки не використовуються —
  потім не доведеться робити міграцію бази.
- Лічильники (views та ін.) — рахувати й зберігати з самого початку.
  Показувати на сайті — ЛИШЕ коли числа достатні (малі числа на старті
  відлякують, сигналять «сервіс мертвий»).
- Лайки — НЕ робимо: для утилітарного сервісу оголошень користі немає.
- Рейтинги/відгуки — корисні для довіри, але складні; Фаза 2.

## 11. Застереження на майбутнє (щоб потім менше переробляти)

- Схема даних — найдорожче міняти потім (міграція бази). Продумати
  поля одразу, до першого коду Фази 1.
- Тексти інтерфейсу — тримати зібрано в одному місці, не розкидати
  по коду. Якщо колись знадобиться інша мовна версія — буде легко.
- Кодування — усі файли UTF-8. Україномовний код писати в редакторі
  (VS Code), не через термінал (PowerShell ламає кирилицю).
- Бот і стару логіку НЕ видаляти, доки Telegram Login на сайті
  не запрацює і не буде перевірений.
- Безкоштовні сервіси (Vercel, MongoDB Atlas, Nominatim) мають ліміти —
  знати їх, не закладатися на безмежність при зростанні.
- Приватність — telegramUserId і телефони це персональні дані.
  Закласти можливість видалити дані користувача на його запит.
- Тестувати на реальному телефоні, не лише у звуженому десктоп-браузері.

## 12. Візуал — узгоджені рішення

- Світла тема: біло-сіра база + зелений акцент (екологічна асоціація).
- Mobile-first — головні користувачі на телефоні.
- Оголошення видно одразу, без прокрутки; без лендинг-секцій.
- Прибрати пафосні тексти («Ексклюзивна спільнота», «Преміальні поїздки»).
- Прибрати секцію «Як це працює».
- Пошук, фільтри і перемикач «Список / Карта» — компактно вгорі.
- Кнопка «Створити оголошення» — зафіксована знизу екрана.
- Прибрати кнопку «У каналі» з карток; лишити «Написати».
- Ілюстрація-фон лише вгорі (ненав'язлива).
- Шрифт — для довіри краще звичайний, не Comic Sans (вирішити остаточно).

## 14. СЕО — основа залучення

СЕО — це основа, не другорядне. Для приміського сервісу працює,
бо великі міста-супутники (Бровари, Ірпінь, Буча, Вишгород,
Бориспіль, Васильків тощо) — це десятки тисяч щоденних поїздок,
і люди гуглять «попутники Бровари Київ» і подібне.

- Базове СЕО — обов'язкове: правильні title, H1, мета-теги,
  sitemap.xml, Google Search Console.
- 10 статичних сторінок міст-вузлів (приміські напрямки) — РОБИМО.
- Внутрішня перелінковка — обов'язково: кожна СЕО-сторінка веде
  вглиб сайту (на головну, карту, споріднені маршрути).
- Локальне поширення теж важливе: канал, спільноти, «Моя спільнота».

Тексти головної — НЕ шаблонні, звірити з реальними запитами через
Google Keyword Planner. Орієнтовні слова: «попутники», «попутчики»,
«блаблакар», «попутка», «попутники [місто-супутник]».
Остаточні title / H1 / опис — ПІСЛЯ перевірки Keyword Planner.

## 15. Технічні нотатки

- Перевірити на Vercel змінну NEXT_PUBLIC_CHANNEL_USERNAME = poputtky_ua
  (дві «t»). У коді є fallback з однією «t».
- Бот на JustRunMy: за Шляхом 2 бот-контейнер більше не потрібен.
- Карта в Next.js підключається окремо, клієнтський рендер, lazy-load.
- Координати — формат GeoJSON з самого початку (для майбутнього 2dsphere).
- Реєстр населених пунктів — окрема колекція в MongoDB або файл даних;
  автокомпліт через власний серверний endpoint.
- SSR Next.js робить за замовчуванням — окремо «робити SSR» не треба.

### 🔴 Пріоритет 1 — Критично (зробити якнайшвидше)

#### 16.1. Архів замість видалення лістингів (SEO: захист від 404)

Проблема: прострочене або видалене оголошення повертає 404 — Google втрачає сторінку.

Рішення:
- Ніколи не видаляти документ фізично. Лише `isActive: false` + `archivedAt`.
- Створити сторінку `/listings/[id]` яка повертає HTTP 200 навіть для архівних,
  показує плашку «Архівне оголошення» + блок «Актуальні поїздки в цьому напрямку».
- Google не отримує 404, користувач не в глухому куті.
- Це вже частково реалізовано (DELETE → isActive: false). Лишилось: роут `/listings/[id]`.

#### 16.2. Поле tripScope: 'suburban' | 'intercity' + фільтр

Дві різних аудиторії:
- `suburban` — приміські щоденні (Бровари → Київ, регулярно, шукають постійного попутника)
- `intercity` — міжміські (Київ → Харків, разово, шукають за датою)

Реалізація:
- Одне поле в MongoDB: `tripScope: "suburban" | "intercity"`.
- У формі: дві кнопки поряд з tripType.
- У фільтрі стрічки: радіо «Всі / Приміські / Міжміські».
- Мінімальна робота, висока цінність для пошуку.

#### 16.3. Карта — UX: клік замість hover + геолокація

Концепція карти (зафіксована):
- Карта — для просторового пошуку («що є поруч від мене?»). Без пошукового рядка.
- Список — для фільтрованого пошуку (за маршрутом, роллю, типом).
- Маркери сірі за замовчуванням → при кліку/тапі стають кольоровими,
  з'являється пунктирна лінія маршруту. Перший тап — «лочить» маршрут,
  другий тап або тап на іншому — переключає.

Що додати:
- Перейти з hover на click/tap (hover не працює на мобільному).
- Кнопка 🎯 «Де я зараз» (геолокація) — центрує карту на поточній позиції.
- Bottom sheet замість popup — на мобільному плашка знизу зручніша за спливаюче вікно.

10. Бориспіль аеропорт (специфічний, окремо)

На кожній сторінці: опис маршруту, зупинки, номери маршруток/електричок,
актуальні лістинги. SEO-магніт: «маршрутка Бровари Київ зупинки».

Реалізація: статичний JSON з маршрутами/зупинками + рендер на `app/city/[slug]/page.tsx`
(вже існує). Одноразова робота, довгострокова цінність.

#### 16.6. Адмін-панель /admin

Мінімум:
- Перегляд усіх лістингів (активні / архів) з фільтрами і пошуком.
- Ручне архівування / відновлення лістингу.
- Список користувачів + дата реєстрації.
- Блокування користувача (антиспам).
- Проста статистика: кількість лістингів/день, кількість юзерів.

Реалізація:
- Захищений роут `/admin` з перевіркою `role === "admin"` у JWT.
- Проста таблиця MongoDB через існуючий clientPromise.
- Потрібна з першого дня активного використання.

### 🟢 Пріоритет 3 — Після MVP-валідації

#### 16.7. Facebook Page автопостинг

- Кожне нове оголошення автоматично публікується на Facebook Page проєкту.
- Meta Graph API: постинг на сторінку бізнесу — реально і відносно просто.
- Постинг у групи через API — обмежений після Cambridge Analytica (потрібен ручний approve).
- Рекомендація: Telegram-канал + Facebook Page = два канали дистрибуції.
- Реалізувати після того як Telegram-канал стабільно працює.

#### 16.8. Email magic link (Resend.com)

- Альтернатива Telegram Login для тих хто не має Telegram.
- Resend.com: безкоштовно до 100 листів/день, легка інтеграція.
- Флоу: вводиш email → отримуєш посилання → кліком входиш.
- Реалізувати якщо Telegram Login залишатиметься нестабільним.

#### 16.9. Instagram автопостинг

- Лише через Meta Business API, складніше ніж Facebook.
- Відкласти до стабільного росту аудиторії.

### 📋 Холодний старт — що реально допомагає

1. **Контент одразу** — сторінки міст із маршрутами/зупинками дають органічний
   трафік до появи першого лістингу.
2. **Посівні оголошення** — кілька оголошень від тебе або знайомих на старті.
3. **Архівні лістинги як SEO** — прострочені оголошення залишаються видимими
   з плашкою «Архів». Google індексує, користувач бачить активність.
4. **Telegram-канал як буфер** — може набирати підписників до запуску сайту.
   Ручний щоденний постинг «шукаю попутника Бровари→Київ» на старті.
5. **Простота публікації** — чим менше кроків до першого оголошення,
   тим менший холодний старт. Мета: 3 кроки від відкриття сайту до опублікованого оголошення.
